这不是对遥远未来的预测。单个 Agent 的长程工作、少量 Agent 的并行协作、持久化项目状态和人工审批机制都已经开始出现。真正尚未解决的是，如何把这些能力组织成一种可以扩大规模、长期运行，同时不过度消耗人类注意力的通用生产系统。

> **核心主张：真正需要扩展的不是 Agent 数量本身，而是在极低人类带宽下，持续完成、验证并整合"可靠工作闭环"的能力。**

## 为什么这个问题重要

Coding Agent 是当前最核心、也最具外溢性的 Agent 形态之一。越来越多的知识工作与工程流程最终都落在软件上：理解需求、读写代码、操作数据、调用工具、构建评测、自动化流程并根据反馈继续迭代。一个能够稳定完成这些工作的 Agent，不仅提高软件生产效率，也能够进入研究、产品、运营和复杂工程流程，并参与开发其他 Agent、工具与自动化系统。

即使最终目标发生在物理世界，Coding Agent 也可以通过开发更可靠的感知、规划、控制、仿真和机器人算法，间接扩大自动化能够覆盖的范围。生物与医疗是更具体的例子：问题本身在物理世界，但大量工作仍要通过计算模型与软件管线来完成；若能把类似 AlphaFold 这类模型的研发速度加快十倍、百倍，便可能实质性推动结构生物学、药物发现等方向的发展。它不会消除硬件、实验和现实约束，但已经能够加快部分系统的设计、验证和迭代。Research Agent 可以看作其中更开放、更高不确定性的一类工作，而不是这套构想的边界。

因此，值得讨论的并不是"某一天 Agent 会不会多写一些代码"，而是已经出现的这些能力能否被组织成高带宽、可验证、可持续的工作过程。若这种组织方式能够可靠扩展，单位时间产出出现数量级提升将成为一个值得认真对待的可能性；它的影响也不会只停留在软件、AI 或研究行业。

## 为什么现在值得讨论

一方面，人类的数量、注意力和工作时长始终有限。另一方面，Agent 的有效供给和能力正在同时受益于四类进展：基础模型能力、Agent harness 与工具、推理和训练算力，以及执行基础设施。更强的模型可以处理更复杂的任务；更好的 harness 能让同一模型更可靠、工作更久；更成熟的 infra 则让更多任务并行，并让结果更快回流。

这个过程还已经开始形成正反馈。Agent 的开发很大一部分本身就是写代码、设计评测、分析失败和优化系统；Coding Agent 因而能够直接参与改进模型训练、工具、harness、基础设施和下一代 Agent。随着这一循环加速，瓶颈也开始从"单个 Agent 是否足够聪明"转向"我们是否知道怎样组织越来越多、越来越强的 Agent"。

这些元素已不再停留在概念层面。Fable 5 等近期系统显著拉长了单 Agent 的工作周期，数天级无人值守工作已开始进入现实能力范围。少量 Agent 并行、持久化状态、自动验证和人工审批也已出现。尚未解决的是，如何让它们在同一系统中随着 Agent 数量和时间扩大，而不让重复、验证、交付整合、人类审批或状态污染成为新瓶颈。

## 重新定义 Scaling 的对象

不宜把"Agent 数量"或"总 token 数"直接当作最终扩展目标。100 个 Agent 同时生成代码、方案或报告，并不自动等于 100 倍生产力。更有价值的基本单元，是一个完整、可验证、能够被后续工作复用的工作闭环：

**理解目标 → 分解与规划 → 执行 → 验证 → 交付并整合**

一个有用的抽象是把系统的累计有效产出写成 Q(N, T, H, B)：N 是并行 Agent 数量，T 是持续运行时间，H 是人类注意力，B 是模型调用、计算、工具和执行资源。我们希望研究的是，怎样让 Q 随 N 和 T 继续增长，同时让 H 增长得尽可能慢。换句话说，需要扩展的不是"对话窗口"，而是单位时间内完成、验证并整合多少个可靠工作闭环。

| **问题维度** | **核心问题** | **希望拉高的曲线** |
| --- | --- | --- |
| 人类边界 | 人类只保留不可外包的目标、价值与高风险决策 | 人力不随 Agent 数量和时间近线性增长 |
| 横向扩展 | 更多 Agent 同时推进互补工作 | 加速比随 N 仍保持正增长，而非 3–4 个后饱和 |
| 时间扩展 | 系统连续运行后仍能产生新增价值 | 长期产出率下降更慢，能够稳定工作数天乃至更久 |

## 第一个问题：人类与 Agent 的工作边界

若要继续提高 Agent 系统的上限，人类就需要逐步退出日常执行的 common path。只要每个任务、每次合并、每次交付或每个结论都需要人确认，人类吞吐量就仍然是系统的硬约束。一种更可扩展的安排，是让人类主要位于低带宽的目标与治理控制层，并只处理少量真正需要升级的异常。

这里可以区分两类"需要人"的原因。第一类是主权性决策：工作目标、价值取舍、风险预算、不可逆的现实行动、是否改变成功标准，以及是否部署、发布或对外承诺重要结果。这些事情即使 Agent 在技术上能够完成，人类也可能仍需承担最终责任。第二类来自当前的能力差距，例如 root-cause diagnosis、关键验证设计、判断异常是否值得追，或解决相互冲突的证据。随着 Agent 能力变化，这一部分边界也应当被持续重新评估。

因此，人类与 Agent 的边界不一定适合按固定流程写死成"人给目标、Agent 执行、人再逐项验收"。一种更值得探索的方式是按状态升级：当任务明显超出已知分布、Agent 之间高度分歧、操作成本或不可逆性很高，或者关键结果无法验证时，再调用人类；其余环节尽量由 Agent 自主闭环。哪些状态真正需要人，应该成为可以被实证研究的问题，而不是预先假定的答案。

> **理想目标：Agent 数量扩大十倍、运行时间扩大十倍时，人类投入最好几乎保持不变。人类应成为控制面，而不是数据面。**

## 第二个问题：效率如何随 Agent 数量增长

在不少现有设置中，multi-agent 系统在少量 Agent 之后就出现明显的边际收益下降。原因往往不是缺少更多名义上的 worker，而是缺少更多独立且有价值的工作通道。相同模型、相同上下文和相同目标的 Agent 容易产生同质方案；与此同时，任务执行、结果验证和产物整合的速度也常常跟不上 proposal 的增长。

一种值得重点探索的组织形式，是动态 Agent pool，而不是固定团队。系统长期保存的是工作状态——目标与任务图、假设和约束、证据、代码与产物谱系、未解决矛盾和资源状态；Agent 则可以是临时的计算单元。它们被创建，获得有限任务和必要上下文，在隔离环境中工作，提交结构化产物，然后在任务完成后退出。

**发现可并行任务 → 动态创建 Agent → 独立执行 → 验证与复现 → 整合并重新调度**

这个 pool 若只扩展"提出方案"的 Agent，通常很快会遇到新的瓶颈。一个生产系统至少包含 planning/proposal、execution、verification 和 integration 四类能力，而系统吞吐量受最慢的一环约束：如果 100 个 Agent 每小时提出 500 个改动，而 verifier 只能处理 10 个，其余规模主要会制造积压、冲突和噪声。一个更合理的 scheduler 需要根据实时队列和瓶颈，动态增加 explorer、executor、verifier 或 integrator，并暂停、迁移或销毁低价值分支。

1. **发现并行性。** 复杂工作一开始往往不是 100 张独立工单，系统必须主动把模糊目标展开成动态任务 DAG，并持续扩大当前可并行的 frontier。
2. **制造非冗余。** 名义上的 N 不重要，真正重要的是能够提供独立方案、不同证据和互补技能的有效 Agent 数 N_eff。
3. **稀疏协作。** 100 个 Agent 不能彼此阅读全部日志。协作应通过任务依赖、结构化 artifact 和证据发生，而不是通过全员群聊。
4. **同步扩展验证与整合。** 生成候选只是前半程。测试、复现、冲突消解、分支合并和交付更新必须拥有同等规模的带宽。
5. **学习组织方式。** 系统需要跨任务记录哪些 Agent、角色组合和分工产生了真实进展，从而让资源调度本身不断变好。

我们关心的曲线，是单位时间内的有效工作产出如何随 N 增长。比如，"100 个 Agent 在一天内完成单 Agent 约 40 天的有效工作"可以作为一个有解释力的目标，而不是对当前能力的描述。评测时还必须区分真实协作收益、独立尝试后选最好结果，以及单纯增加总算力。只有协作系统在可比预算下持续优于独立工作，才能说明组织本身产生了 1+1>1 的价值。

## 第三个问题：效率如何随运行时间保持

Fable 5 等近期系统已经把单个 Agent 能够连续工作的时间显著推长。对于目标相对明确的任务，数天级无人值守推进正在成为现实。因此，这一部分不需要再把重点放在"如何让 Agent 一直运行"或具体的技术实现上。

我们真正关心的是另一件事：工作时间延长以后，产出是否仍然有价值。一个 Agent 连续运行几天，可能不断取得进展，也可能只是重复尝试，或者缓慢消耗一个方向上剩余的问题。Coding 和 research 是其中要求最高的例子，但这个问题同样适用于产品开发、运营和复杂工程：重要的不是 Agent 工作了多久，而是在更长的时间尺度上仍能带来多少新的、可用的结果。

所以要拉高的是"单位时间有效产出随运行时间变化"的曲线。理想状态不是让 Agent 永远不停，而是让它在更长时间内保持较高产出，并能在继续、转向或结束任务之间做出合理选择，尽量不依赖人类持续看守。Fable 5 表明这条曲线已经在被向外推；接下来要观察的是，这种进步能否从单个长任务扩展到开放式、连续的复杂工作。

## 把三个问题放在一起：可扩展的自主工作系统

这三个问题彼此耦合。更多 Agent 会更快制造协调和整合债务，从而压低时间曲线；运行时间越长，异常和冲突越多，也越容易把人重新拉回 critical path。因而，真正需要扩展的不是某一条孤立曲线，而是由 Agent 数量 N、持续时间 T 和人类注意力 H 共同决定的"可扩展自治区域"。

> **人类设定目标、价值与风险边界 → 持久化项目状态与动态调度器 → 可创建/销毁的 Planner、Explorer、Executor、Verifier、Integrator → 经验证的代码、知识、决策与下一轮任务**

这套构想倾向于一种正在形成的组织方式：人类负责低带宽的 constitution 和 exception；持久存在的是项目与工作状态；Agent 是可以弹性创建和销毁的计算单元；协作主要通过 artifact 和证据发生；系统根据当前瓶颈动态改变 worker 构成；结果只有在被验证、交付并整合进后续流程后，才算真正形成有效产出。

进一步看，长期稳定产出很可能还需要任务与项目组合，而不是把 100 个 Agent 始终放在同一个问题上。单个方向的低垂果实会逐渐耗尽，组织需要在多个项目之间重新配置资源：饱和的方向减员，出现异常或新机会的方向增员，等待外部结果的方向暂停，并围绕新发现的问题快速组建临时团队。

## 这套构想指向什么

这一变化已经开始发生。决定工作速度的因素正在从"最强模型能做什么"，扩展到"围绕模型的生产系统能组织什么"：人类判断是否只出现在真正高价值的位置，大量 Agent 是否能同时推进互补工作，以及项目状态能否在长时间运行中保持目标、证据、产物和版本的一致性。

如果横向曲线能够从少量 Agent 后的快速饱和，推向几十甚至上百个 Agent 仍有可观边际收益；如果时间曲线能够从数小时的高效窗口，稳定延伸到数天乃至更久；同时人类注意力不按同样速度增长，那么知识工作与复杂工程的单位时间产出就有可能获得数量级提升。Coding Agent 是其中最核心的底层杠杆：软件连接了研究、产品、业务系统和机器人开发，因此这种组织能力可以从数字工作逐步辐射到制造、能源、生物、材料以及更多现实行业。

> **一种正在出现的通用生产力单元，是由持久化项目状态、弹性 Agent、有限人类治理和可验证工作闭环组成的自主工作系统。**
